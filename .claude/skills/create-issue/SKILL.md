GitHub Issue를 **타입에 맞는 본문까지 작성해서** 생성하는 스킬. (저장소: `hs-shell/dotbugi`)

오픈소스 이슈는 "구현 로드맵"이 아니라 **리포트/요청**이다. 타입별로 `.github/ISSUE_TEMPLATE/`의 구조를 그대로 따른다. 구현 계획은 이슈가 아니라 PR에서 다룬다.

## 1단계: 타입 결정

사용자 설명에서 타입을 추론하고, 애매하면 묻는다:

| 타입                       | 라벨                                    | 제목 prefix |
| -------------------------- | --------------------------------------- | ----------- |
| 🐛 버그                    | `🐛 bug`                                | `[Bug] `    |
| ✨ 기능 제안               | `✨ feat`                               | `[Feat] `   |
| 📝 문서                    | `📝 docs`                               | `[Docs] `   |
| 🧹 작업(chore/ci/refactor) | `🧹 chore` (또는 `⚙️ ci`/`♻️ refactor`) | `[Chore] `  |

- 제목은 서술형(한국어/영어 허용). 이슈 제목엔 Conventional Commits(`fix:` 등)를 쓰지 않는다 — 그건 PR/커밋용.

## 2단계: 타입별 본문 작성

`.github/ISSUE_TEMPLATE/`의 필드 구조를 따라 사용자와 함께 채운다.

- **버그** (`bug_report.yml` 기준): 무슨 일이 일어났나 / 재현 방법 / 기대한 동작 / 확장 버전 / 브라우저·OS / (선택) 스크린샷·콘솔 로그
- **기능 제안** (`feature_request.yml` 기준): 어떤 문제를 해결하나 / 제안하는 해결책 / 고려한 대안
- **문서·작업**: 배경(왜 필요한지) + 할 일 체크리스트

확실치 않은 칸은 비워두지 말고 사용자에게 물어 채운다.

## 3단계: 생성

```
gh issue create --repo hs-shell/dotbugi \
  --title "<prefix><설명>" \
  --label "<타입 라벨>" \
  --body "<작성한 본문>"
```

- 담당자가 필요하면 `--assignee` 추가.

## 4단계: 결과

생성된 이슈 번호와 URL을 출력한다.

- 다음 액션 안내: 직접 작업할 거면 `/implement <번호>`, 남이 올린 이슈를 분석/검증할 거면 `/analyze-issue <번호>`.
