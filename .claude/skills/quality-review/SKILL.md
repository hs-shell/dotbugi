description: 사용자가 "품질 리뷰", "보안 안정 확장 성능 점검", "구현 보강", "취약점 점검해서 반영", "고칠 부분 없는지 봐줘" 등을 요청하거나, 현재 구현을 보안·안정성·확장성·성능 관점에서 점검해 반영해야 할 때 사용

현재 구현(브랜치 diff)을 **보안·안정성·확장성·성능** 네 관점에서 점검하고, 타당한 지적을 검증해 코드에 반영하는 스킬.

점검은 기존 스킬을 재사용한다 — 중복 구현하지 않는다.

## 1단계: 변경 범위 파악

```
git diff main...HEAD --stat
git diff main...HEAD
```

- 변경이 없으면 알리고 종료. PR이 있으면 PR diff 기준으로 봐도 된다.

## 2단계: 점검 (기존 스킬 재사용)

| 관점   | 사용 스킬           | 비고                                      |
| ------ | ------------------- | ----------------------------------------- |
| 보안   | `/security-review`  | injection, 권한, 비밀정보 노출, 입력 검증 |
| 안정성 | `/code-review high` | 정확성 버그, 엣지케이스, 에러 처리, 회귀  |
| 성능   | `/code-review high` | 불필요한 재계산, 메모리, 비효율 경로      |
| 확장성 | 본 스킬에서 보강    | 결합도, 하드코딩된 한계, 인터페이스 경계  |

- `/code-review`는 `--fix` 없이 호출해 findings만 받는다.
- 확장성 직접 점검 시 dotbugi 함정을 본다: LMS HTML 파싱의 셀렉터/키워드 의존성(LMS 개편 시 깨짐), 캐시 TTL·키 생성기(`makeVodKey` 등) 가정, Shadow DOM 격리 경계, 메시지 패싱 계약.

## 3단계: 타당성 검증

수집한 findings를 코드 맥락에서 검증한다.

- **타당함** → 반영 대상.
- **오탐/스코프 밖/트레이드오프 큼** → 보류, 근거 기록.
- 추측 금지 — 실제 코드·동작으로 확인.

## 4단계: 반영 (검증 후 자동 반영)

- 변경 코드에 해당하는 `.claude/rules/` 준수.
- 수정 후 CI 동일 조건 검증(`common.md`: `npm run lint` / `npm run build` / `npm run test`).
- 보류 항목은 반영하지 않는다.

## 5단계: 보고

관점별·항목별 한 줄: **반영 / 보류(사유)**. 사용자 판단이 필요한 트레이드오프는 명시 질문.

## 주의

- "현재 구현 보강"이 목적 — 스코프 밖 대규모 리팩토링은 별도 이슈로 분리 제안(1 PR = 1 Issue).
- 변경이 크면 관점별로 나눠 커밋한다.
